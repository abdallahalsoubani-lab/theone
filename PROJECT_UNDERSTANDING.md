## 1. What this is

**Theone.pt** is a bilingual (English + Arabic, RTL-first) physiotherapy
clinic management web application built specifically for **The One for
Physiotherapy** (المركز الأول للعلاج الطبيعي) in Amman, Jordan. It is a
**single-clinic** operational tool (multi-tenant is explicitly v2), sized
for 5–10 staff and a few hundred active patients.

The product covers the complete patient journey end-to-end: registration →
intake assessment (adult or pediatric) → treatment plan authorship →
appointment scheduling → SOAP session notes → home exercise program →
compliance tracking → optional PDF export. **WhatsApp is the primary
patient communication channel** (confirmations, 30-minute reminders, OTP
login, home-exercise prompts, low-compliance nudges) via a provider
abstraction that supports Console (dev), Twilio Sandbox/Production, and
Meta Cloud API interchangeably.

The system is **feature-complete** as v1.0. It was built across 12
sequential prompts (Prompts 0–12) per the README/CHANGELOG, with the
canonical product spec sitting in `docs/Theone-pt-Technical-Spec.md`
(~63 KB). At the time of writing: **912 tests passing, 1,087 i18n keys
EN+AR in sync, 8 Prisma migrations, 26 models, 28 enums, ~70 permission
codes, 5 roles.**

## 3. The 5 user roles and what each can do

Verified against `lib/rbac/permissions.ts` (the `ROLE_PERMISSIONS` map):

| Role          | Arabic          | Home route             | Core responsibilities                                                                                                                                                                                                                        |
| ------------- | --------------- | ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **PATIENT**   | المراجع         | `/patient/dashboard`   | View own appointments + home program, mark exercises complete, see in-app notifications, edit non-clinical profile fields, receive WhatsApp reminders.                                                                                       |
| **SECRETARY** | السكرتيرة       | `/secretary/calendar`  | Calendar CRUD + drag-reschedule + change-therapist + cancel-with-category + recurring series. Patient registration + intake. Inbox triage (inbound WhatsApp + delivery failures + leave conflicts). May override conflicts (logged).         |
| **DOCTOR**    | الدكتورة        | `/doctor/dashboard`    | Author / approve / reject / pause / discontinue treatment plans for assigned patients. Read SOAP notes for assigned patients. Write weekly DoctorReview entries. Add session-note addenda.                                                   |
| **THERAPIST** | المعالج الطبيعي | `/therapist/dashboard` | Execute treatment plans, write SOAP session notes (immutable after 24h except via addendum chain), submit DayReport, build home-program items, upload exercise media. May propose plan changes (Doctor approves/rejects).                    |
| **ADMIN**     | المسؤول         | `/admin/dashboard`     | User / Specialty / Room / IntakeCustomQuestion / ClinicSettings CRUD. WhatsApp template registry + message log. AuditLog viewer + CSV export. Read-only across clinical content (never writes notes / plans). Reset patient/staff passwords. |

Three permission-scope variants exist (see `lib/rbac/can.ts`):

- **Unscoped** — `appointments.update` etc.
- **Owner-scoped** — `appointments.read.own` — code + `resource.ownerId === user.id`.
- **Assignment-scoped** — `patients.read.assigned` — code + `resource.assignedClinicianIds.includes(user.id)`.

Plus one **override** code that allows knowingly violating a conflict
warning: `appointments.override_conflict` (Secretary + Admin only).

## 4. The data model

Grouped from `prisma/schema.prisma` (26 models, 28 enums):

### Core identity & people

- **`User`** — every human in the system. Carries `role`, bilingual names
  (`fullNameEn` / `fullNameAr`), password + lockout + OTP fields,
  WhatsApp reachability flags. Soft-delete via `deletedAt` (User only).
- **`PatientProfile`** — 1:1 extension of `User` (`userId @id`) holding
  DOB, gender, national ID, address, emergency contact, medical history,
  `hijriCalendarPref`, plus `assignedTherapistId` and `responsibleDoctorId`.
- **`Specialty`** + **`UserSpecialty`** — clinic-wide specialty taxonomy
  (M:N to User).
- **`Room`** — physical treatment rooms; referenced (optionally) from
  Appointment.

### Scheduling

- **`Appointment`** — single source of bookings. Carries patient,
  therapist, room?, `startsAt` (UTC), duration, status, optional
  `cancellationCategory` + `cancellationReason` + `cancellationNotes`,
  `seriesId` (group key for recurring series), createdById.
- **`Leave`** — staff leave requests (SICK / VACATION / PERSONAL).
  PENDING → APPROVED / REJECTED. Approved leaves drive the calendar
  overlay and raise `LEAVE_CONFLICT` InboxItems.

### Clinical content

- **`TreatmentPlan`** — Doctor-authored prescription. Versioned via
  `parentPlanId` chain. Status state machine
  `ACTIVE / PROPOSED / REJECTED / SUPERSEDED / PAUSED / COMPLETED / DISCONTINUED`.
  Carries `proposalReason`, `rejectedReason`, `approvedAt` + `approvedById`.
- **`PlanExercise`** — line items on a plan (M:1 to Exercise) with
  sets / reps / duration / order.
- **`Exercise`** — clinic-wide exercise library. Versioned via
  `replacedById` (1:1 chain). Bilingual name / description /
  default instruction. Optional `videoUrl` + `imageUrl` (via S3).
- **`SessionNote`** — SOAP notes per appointment (`subjective`,
  `objective`, `assessment`, `plan`, `painScore`, `measurements: Json?`).
  Partial unique index enforces at most one **primary** note per
  appointment; addenda share the appointment via `parentNoteId`.
  Immutable after 24h except via addendum chain.
- **`DayReport`** — Therapist's end-of-day summary; `@@unique(therapistId, date)`.
- **`DoctorReview`** — Doctor's weekly per-patient clinical commentary
  (`weekStarting`).

### Home program

- **`HomeProgramItem`** — per-patient exercise schedule. `daysOfWeek: Int[]`
  (0–6, Sun–Sat), `scheduledTime: "HH:MM"` (recurring local-time concept),
  `durationMinutes`, `setsReps?`, `therapistNote?`, `reminderJobKey?`
  (BullMQ repeat-job key for cancellation), `active`.
- **`HomeProgramCompletion`** — per-item-per-day completion log;
  `@@unique(itemId, scheduledDate)`. Drives compliance widgets.

### Intake

- **`IntakeAssessment`** — parent row with `type: ADULT | PEDIATRIC`,
  status, optional reviewer.
- **`AdultIntakeData`** — typed columns for adult-specific fields
  (pain timing, severity, duration, stability, conditions,
  referralSource, previous fractures / surgeries / PT, etc.).
- **`PediatricIntakeData`** — pediatric-specific fields (siblings, birth order).
- **`IntakeCustomQuestion`** + **`IntakeCustomAnswer`** — clinic-configurable
  custom questions with bilingual labels, applyability scope (ADULT / PEDIATRIC / BOTH),
  type (TEXT / TEXTAREA / SINGLE_SELECT / MULTI_SELECT / NUMBER / DATE).

### WhatsApp

- **`WhatsAppTemplate`** — provider-agnostic template registry with
  logical `name` + per-provider metadata (`metaTemplateName`,
  `metaApprovalStatus`, `twilioContentSid`, `twilioApproved`).
- **`WhatsAppMessage`** — provider-agnostic message log. Holds direction,
  status, `providerMessageId`, `intent` (inbound classification),
  `appointmentId` for correlation, `failureReason`, `resendCount`,
  `sentAt` / `deliveredAt` / `readAt`.

### Triage & notifications

- **`InboxItem`** — Secretary's triage queue (inbound reschedule /
  cancel / unknown, outbound delivery failures, leave conflicts).
- **`Notification`** — generic in-app notification with i18n key
  templates (`titleKey`, `bodyKey`) + `params: Json` for ICU
  interpolation. 15 `NotificationType` values; see
  `lib/notifications/templates.ts → NOTIFICATION_TEMPLATES`.

### Audit & settings

- **`AuditLog`** — append-only `{ actorId, entityType, entityId, action, before, after }`.
  Indexed on `(entityType, entityId, createdAt desc)` + `(actorId, createdAt desc)`.
- **`ClinicSettings`** — singleton (`id = "default"`). Holds clinic
  name / phone / addresses (bilingual), timezone (default `Asia/Amman`),
  default appointment duration + reminder offset, **`businessHours: Json`**
  (the conflict engine reads this directly), `defaultLanguage` (default `AR`),
  `serviceTypes: Json`, `hijriDefault`, `patientCanViewClinicalNotes` (default false).

## 5. The patient journey (end-to-end)

References to prompts are taken from `CHANGELOG.md` / `CLAUDE.md`.

1. **First contact / Registration** — Secretary registers the patient
   via `/secretary/patients` (Prompt 6). The action provisions a
   `User` with `role: PATIENT` + `PatientProfile`, generates a temp
   password (`lib/admin/temp-password.ts`), and queues a
   `patient_account_credentials` WhatsApp template with the login info.
2. **Intake** — Secretary opens the patient file → Intake tab
   (`components/patients/PatientIntakeTab.tsx`), picks ADULT or
   PEDIATRIC, fills the form (`components/intake/{AdultIntakeForm,PediatricIntakeForm}.tsx`).
   Custom questions from `IntakeCustomQuestion` are rendered via
   `CustomQuestionField.tsx`. Saved as `IntakeAssessment` + detail row(s)
   (Prompt 6).
3. **Treatment plan** — Doctor opens the patient file → Plan tab
   (`components/clinical/PatientPlanTab.tsx`) and authors a
   `TreatmentPlan` with diagnoses, goals, frequency, duration, and
   exercises drawn from the Exercise library. Plan starts `ACTIVE`
   immediately or `PROPOSED` if a Therapist authored a revision (Prompt 9).
4. **Appointment booking** — Secretary opens
   `/secretary/calendar` (`components/calendar/SecretaryCalendar.tsx`),
   drags on an empty slot or clicks "+ New". `CreateAppointmentModal`
   submits to `createAppointment` in `lib/appointments/actions.ts`,
   which:
   1. checks permission (`requirePermission('appointments.create')`),
   2. validates input (`appointmentCreateSchema`),
   3. runs `checkConflicts` (Prompts 7 + 7b),
   4. inserts the row inside `withAudit`,
   5. enqueues a 30-min reminder onto the `reminders` queue.
      For recurring series the Secretary uses `CreateSeriesModal` → up to
      52 weekly occurrences with per-occurrence resolution (Skip / Shift +1d
      / Shift +1w / Override) — all inserted transactionally.
5. **WhatsApp confirmation** — `workers/whatsapp.ts` picks up the
   `appointment_confirmation` template (EN or AR per patient
   `languagePref`), sends via the active provider, records the
   `WhatsAppMessage` row, and on failure raises an
   `OUTBOUND_DELIVERY_FAILED` InboxItem (Prompt 8).
6. **30-min reminder** — `workers/reminder.ts` consumes the queued job
   and fires `appointment_reminder_30min` (Prompt 7).
7. **Session** — Therapist marks the appointment `IN_PROGRESS` then
   `COMPLETED`, which opens the SOAP form
   (`components/clinical/SessionNoteForm.tsx`). The note is the
   primary `SessionNote`; later edits past 24h require an addendum
   (Prompt 9).
8. **Home program** — Therapist builds it via
   `components/home-program/HomeProgramBuilder.tsx`. Each
   `HomeProgramItem` registers a BullMQ **repeat-job** on the
   `reminders` queue (job name `homeExerciseReminder`) with a cron
   derived from `daysOfWeek` + `scheduledTime − HOME_REMINDER_OFFSET_MINUTES`,
   `tz: Asia/Amman` (Prompt 10).
9. **Compliance** — Patient marks items done via
   `components/home-program/MarkAsDoneButton.tsx`. Compliance widgets
   on the patient file (7-day + 30-day, streak, last completion) are
   computed in `lib/clinical/compliance/`.
10. **Low-compliance nudge** — `workers/complianceDailyCheck.ts`
    runs daily at 18:00 Asia/Amman, scans active home programs,
    and emits `LOW_COMPLIANCE` Notification rows to the responsible
    therapist for patients < 50% adherence over 7 days (Prompt 10).
11. **PDF export / right-of-access** — Patient or Admin clicks
    "Download my file" → `app/api/v1/exports/patient-file` →
    `lib/exports/patientFile.tsx`. Role-aware redaction via
    `lib/exports/redaction.ts → resolveRedaction`. Audited as
    `READ_SENSITIVE` (Prompt 11).
12. **Inbound replies** — Patient replies `1` / `confirm` /
    `reschedule` etc. on WhatsApp. Webhook lands at
    `app/api/v1/whatsapp/webhook/{twilio,meta}/route.ts`,
    signature-verified, classified via `lib/whatsapp/inbound/parser.ts`,
    creates an `INBOUND_*` InboxItem for the Secretary (Prompt 8).

## 7. Key abstractions and where they live

- **Conflict engine** — `lib/appointments/conflicts.ts → checkConflicts()`.
  Five conflict kinds: `THERAPIST_OVERLAP`, `PATIENT_OVERLAP`,
  `THERAPIST_ON_LEAVE`, `OUTSIDE_BUSINESS_HOURS`, `CLINIC_CLOSED_THIS_DAY`.
  Exhaustive (returns every applicable conflict, not first-match).
  Reads `ClinicSettings.businessHours` JSON for hours / closed-days.
- **Series operations** — `lib/appointments/services.ts →
cancelAppointmentSeries / rescheduleAppointmentSeries /
changeAppointmentTherapistSeries` + `lib/appointments/recurrence.ts`
  (weekly expansion). All-or-nothing transactional with
  `BulkAppointmentError`. `seriesMode: 'ONE' | 'FOLLOWING' | 'ALL'`
  is **always explicit**, never inferred.
- **WhatsApp providers** — `lib/whatsapp/index.ts` (the factory; **note:
  CLAUDE.md says `factory.ts`, the actual file is `index.ts`**) +
  `lib/whatsapp/providers/{console,twilio,meta}.ts`. The factory reads
  `env.WHATSAPP_PROVIDER` and pins the instance on `globalThis` for
  HMR safety. Callers never branch by provider name.
- **WhatsApp outbound queue** — `lib/queue/jobs/whatsappOutbound.ts` +
  `workers/whatsapp.ts`. Single chokepoint for **every** outbound
  message (OTP, credentials, confirmations, cancellations, reminders,
  resends).
- **Inbound parser** — `lib/whatsapp/inbound/parser.ts` (EN + AR
  keywords for confirm / reschedule / cancel) + `lib/whatsapp/inbound/process.ts`.
- **Auth + RBAC** — `lib/auth/config.ts` + `lib/auth/providers.ts` for
  Auth.js; `lib/rbac/permissions.ts` (catalog), `lib/rbac/can.ts`
  (resolver), `lib/rbac/guards.ts` (`requirePermission` for actions).
- **Audit log** — `lib/audit/withAudit.ts`. Wraps every state-changing
  service function. Failures to write the audit row are caught and
  logged to stderr (never break the user operation).
- **Notifications** — `lib/notifications/templates.ts → NOTIFICATION_TEMPLATES`
  (one row per `NotificationType`) + `lib/notifications/actions.ts` +
  `lib/notifications/queries.ts`. UI in `components/notifications/`.
- **i18n** — `messages/{en,ar}.json` (1,317 lines each) + `i18n/routing.ts`
  - the next-intl middleware in `middleware.ts`. CI gate:
    `scripts/check-i18n-sync.ts` via `pnpm i18n:check`.
- **Storage** — `lib/storage/presign.ts` (presigned PUT URLs) +
  `lib/storage/client.ts` (S3 client) + `lib/storage/policies.ts`

## 8. Conventions to follow when modifying this codebase

Distilled from `CLAUDE.md` and verified against the actual code:

1. **Server actions return `Result<T, LocalizedError>`** — never
   throw across the action boundary. See `lib/auth/result.ts` for the
   shape; pattern visible in `lib/appointments/actions.ts`.
2. **Wrap mutating services with `withAudit(...)`** — see
   `lib/audit/withAudit.ts`. Forgetting is a security-review blocker.
3. **Gate at the action boundary with `requirePermission(...)`** —
   see `lib/rbac/guards.ts`. UI gating is **not** sufficient.
4. **Forms** = react-hook-form + Zod resolvers. The Zod schema is the
   single source of truth; UI labels come from next-intl. Reference:
   `components/forms/AppForm.tsx`.
5. **Tables** use `components/data-table/DataTable.tsx`. Filter state
   lives in `searchParams`, not React state — refresh-safe + linkable.
6. **One conflict engine** — every appointment create / move routes
   through `lib/appointments/conflicts.ts → checkConflicts()`. Don't
   re-derive availability anywhere else.
7. **Series edits** always pass an **explicit** `seriesMode: 'ONE' |
'FOLLOWING' | 'ALL'`. `FOLLOWING` is boundary-inclusive (`startsAt >=
target.startsAt`). Bulk paths use `db.$transaction` for all-or-nothing.
8. **WhatsApp callers go through the factory** —
   `import { whatsapp } from '@/lib/whatsapp'`. Never branch by provider
   name outside `lib/whatsapp/providers/`.
9. **All outbound WhatsApp goes through the `whatsappOutbound` queue** —
   `lib/queue/jobs/whatsappOutbound.ts` + `workers/whatsapp.ts`. Uniform
   retries + audit + rate limiting.
10. **Notifications** require: enum value in `NotificationType`, row in
    `NOTIFICATION_TEMPLATES`, two i18n keys
    (`notifications.types.{type}.title|body`) in **both** `en.json` and
    `ar.json`.
11. **i18n** — zero hardcoded UI strings. Every label goes through
    `useTranslations('namespace')`. CI gate: `pnpm i18n:check`.
12. **Permissions** — `lib/rbac/permissions.ts` is the catalog. New code
    requires a row update in `lib/rbac/__tests__/can.test.ts`.
13. **Times + dates** — all `DateTime` columns are UTC at the storage
    layer. Display via `lib/format/date.ts` (`formatDate`, `formatTime`,
    `formatDateTime`). Hijri toggle from `User.patientProfile.hijriCalendarPref`.
    BullMQ repeat jobs use `tz: 'Asia/Amman'`.
14. **Storage** — direct browser-to-S3 via presigned PUT URLs from
    `lib/storage/presign.ts`. Never proxy uploads through Next.js
    handlers. Local MinIO uses `forcePathStyle: true`.
15. **Brand tokens locked** — extend semantically (e.g. `brand-textMuted`)
    rather than introducing parallel design tokens.
16. **Logical CSS properties** — `ms-*`, `me-*`, `ps-*`, `pe-*` etc.
    Never `ml-*` / `mr-*` for layout.
17. **Component size cap** — ≤ 200 lines (Prompt 0 §5 rule 11). Split into
    subcomponents or extract hooks beyond that.

## 9. Anti-patterns to refuse

Direct from `CLAUDE.md`:

- `if (provider === 'twilio') { ... }` **outside** `lib/whatsapp/providers/`.
- Re-implementing conflict checks anywhere except
  `lib/appointments/conflicts.ts`.
- Direct `db.X.update(...)` from a client component or a route handler
  that isn't a server action — must go through `lib/{domain}/actions.ts`.
- Hardcoded UI strings in any locale.
- New npm dependencies without justification. The stack is tight.
- Modifying the `brand.*` tokens or introducing parallel design tokens.
- Bypassing `requirePermission(...)` in a server action.
- Logging PII (phone numbers, emails, full names, free-text notes)
  without going through a redaction helper.
- Skipping the `withAudit` wrapper on a state-changing service.
- Adding a `NotificationType` / `InboxItemType` enum value without the
  matching template + i18n keys.
- Inferring a series-edit scope when an explicit `seriesMode` is missing.
- Patching the audit log (it's append-only — no "fix the audit row"
  workflow exists, and none should).

## 14. Things I'm uncertain about

These are gaps, contradictions, or under-documented spots I found between
docs and code during reconnaissance. **This is the highest-signal section.**

1. **`lib/whatsapp/factory.ts` doesn't exist.** Both `CLAUDE.md` and
   `docs/architecture/overview.md` reference `lib/whatsapp/factory.ts` as
   the provider factory; the actual factory lives at
   `lib/whatsapp/index.ts` (lines 26-38, `makeProvider()`). Anyone
   grepping for `factory.ts` will be confused.
2. **Prompts 2–11 are not committed to `docs/prompts/`.** `docs/README.md`
   §"Documents not yet written" flags this explicitly: "only Prompt 1's
   spec made it into `docs/prompts/`. The other 11 prompts exist as the
   project owner's working specifications but aren't checked into the
   repo." So the design-intent docs for the bulk of the system are
   referenced everywhere but not actually present. The only available
   prompt files are `docs/Prompt-0-Master-Context.md` and
   `docs/prompts/Prompt-1-Project-Foundation.md`.
3. **`docs/onboarding.md` references `.env.local.example` but the actual
   file is `.env.example`.** Line 20 of `docs/onboarding.md`:
   `cp .env.local.example .env.local`. The repo has `.env.example`.
4. **Spec § 7.1 lists TanStack Query + Zustand, but neither is
   installed.** `package.json` has no `@tanstack/react-query` and no
   `zustand`. The built system relies on server actions + `router.refresh()`
   and React's `useState` / context — which is consistent with App Router

5. **Spec § 7.1 lists FullCalendar; the implementation uses

6. **The `homeReminder` queue is not a separate queue — it's the same

   attaches to `REMINDER_QUEUE` and filters on `job.name === 'homeExerciseReminder'`.
   Worth confirming whether `HOME_PROGRAM_QUEUE` is actually used or
   left over.
7. **Spec § 7.1 lists Sentry as the error tracker; it's not installed.**
   This is documented as deferred in `docs/backlog.md`, and `lib/logger.ts`

8. **Test counts vs. file counts.** CHANGELOG says "912 tests passing

9. **i18n key count.** CHANGELOG claims 1,087 keys; `wc -l` on
